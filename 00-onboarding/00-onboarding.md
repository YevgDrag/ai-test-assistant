# AI Test Assistant Onboarding

## Мета

Цей документ описує перші кроки для вашого проекту: як перейти від обговорення ідей до першого робочого прототипу.

## Як ми працюємо

Ми будемо йти за підходом з курсу:

1. Починаємо з базової теорії та визначення ролей.
2. Визначаємо мету і бажаний результат проекту.
3. Проектуємо MVP та архітектуру.
4. Переходимо до практики: створюємо робочий прототип.
5. Паралельно з виробничим кодом документуємо кожен крок.

## Послідовність для цього проекту

1. **Огляд ролей та цілей.** Ви з колегою узгоджуєте, хто відповідає за що в проекті.
2. **Написати baseline.** Це те, що вже зроблено, але його треба доповнити: бачення, цілі, принципи, можливості, MVP.
3. **Обрати робочий сценарій.** Наприклад: генерація тест-плану для Jira issue + Confluence вимоги.
4. **Побудувати прототип API.** Простий Python сервіс з endpoint, який приймає Jira/Confluence дані і повертає тест-план.
5. **Вбудувати безпеку.** Створити правила для обробки конфіденційних даних, фільтри і checks.
6. **Перевірити commit flow.** pre-commit, `.gitignore`, secrets scan, document flow.

## Яку роль виконує AI в цьому проекті

- Допомагає формувати текстові артефакти: тест-план, ризики, сценарії.
- Допомагає перевіряти відповідність плану до acceptance criteria.
- Не приймає остаточних рішень про реліз.
- Не обробляє і не публікує конфіденційні дані без попередньої санітизації.

## Що ми ставимо у фокус

- Зрозуміти, яку задачу вирішує продукт.
- Протоколювати, чому ми вибрали саме такий підхід.
- Зробити так, щоб проект було просто пояснити колезі.
- Відразу інтегрувати базові інструменти QA/devops, щоб нічого не загубилося.

## Наступне

Вам варто зараз узгодити з колегою:
- яку роль ви обидва берете в проєкті,
- яку частину проекту хочете зробити першим: baseline, API або інтеграцію,
- які дані з Jira/Confluence будуть вашим входом.

### Ваш setup
- **Мета:** створити AI-асистента для автоматизації QA-процесів, який аналізує вимоги з Jira-таски та Confluence, пропонує список перевірок, питає QA-перевірку на вибір, а потім генерує структурований тест-план у вигляді пунктів.
- **Роль:** ти працюєш як QA інженер.
- **Команда:** PM/BA/PO, дизайнер, фронтенд, бекенд, QA.
- **Перший кейс:** аналіз конкретної Jira-issue і пов'язаної Confluence-сторінки для генерації тест-плану.
- **MVP:** асистент приймає вимоги як текст, аналізує їх і повертає структурований тест-план; перший етап без write-back у Jira.

Після цього я допоможу перейти до конкретного практичного кроку: побудови прототипу, документації або готового робочого flow.
