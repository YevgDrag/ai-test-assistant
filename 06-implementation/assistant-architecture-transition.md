# AI QA Assistant Architecture Transition

## Мета документа

Описати, як наразі працює прототип у `06-implementation/app.py`, і як його логічно еволюціонувати до повноцінного AI-асистента, який генерує тест-плани та QA-пропозиції за допомогою LLM.

## Поточний стан прототипу

### Що вже реалізовано

- `app.py` приймає POST-запит на `/generate-test-plan` та `/generate-test-plan/plain`
- Вхідні дані є у `payload.json`
- У модельному коді (`build_test_plan()`) зараз застосовуються прості правила для формування:
  - `plan_markdown`
  - `suggested_checks`
  - `qa_confirm_question`
- Підтримується два джерела вимог:
  - безпосередньо `confluence_excerpt` у JSON
  - `confluence_file_path` для локального файлу з Confluence-потребами
- `README.md` уже містить приклади запитів та опис endpoint-ів

### Навіщо це працює

Прототип дозволяє нам:
- перевірити API-потік
- зафіксувати очікувану структуру відповіді
- легко замінити правила на AI-генерацію пізніше

## Архітектура переходу до AI-генерації

### Компоненти майбутньої архітектури

1. **Інтерфейс API**
   - залишиться FastAPI
   - збережуться endpoint-и `/generate-test-plan` та `/generate-test-plan/plain`

2. **Збір вхідних даних**
   - `jira_summary`
   - `jira_description`
   - `acceptance_criteria`
   - `confluence_excerpt` або `confluence_file_path`
   - `frontend_changes`
   - `design_available`
   - `additional_context`

3. **Підготовка контексту**
   - конвертація JSON у промпт
   - підстановка тексту таски і Confluence у шаблон з `prompt_template.md`

4. **LLM-генерація**
   - виклик зовнішньої моделі (OpenAI, Azure, локальний LLM тощо)
   - можлива інтеграція через бібліотеку або HTTP-клієнт
   - отримання відповіді у вигляді тексту / markdown / JSON

5. **Парсинг результату**
   - розбір відповіді моделі на такі поля:
     - `plan_markdown`
     - `suggested_checks`
     - `qa_confirm_question`
   - збереження валідованої структури у response model

6. **Повернення відповіді**
   - JSON для `/generate-test-plan`
   - plain text для `/generate-test-plan/plain`

### Базовий data flow

1. Клієнт надсилає `payload.json`
2. Сервер читає payload та, якщо задано, локальний `confluence_file_path`
3. Сервер формує промпт за шаблоном
4. Сервер передає промпт LLM
5. LLM повертає результат
6. Сервер повертає кінцевий тест-план користувачу

## Заплановані зміни у коді

### Наразі

- `build_test_plan()` реалізує правила у Python
- `load_confluence_text()` читає локальний Confluence-файл

### Далі

- замінити `build_test_plan()` на `build_prompt()` + `call_llm()`
- зберегти поточну структуру відповідей
- додати обробку помилок моделі (невірний формат, timeout, пустий результат)
- зробити endpoint-і незалежними від конкретної моделі

## Варіант переходу

### Варіант 1: Найпростіший

- зберегти поточну API-структуру
- у `build_test_plan()` вставити виклик LLM
- поки що залишити `normalize`/fallback правила на випадок поганої відповіді

### Варіант 2: Більш гнучкий

- винести генерацію промпту у окремий модуль
- зробити adapter для різних моделей
- додати feature flags для вибору режиму: `rule-based` / `LLM-based`
